College Project Brief

Real Estate CRM for Indian property teams

This page is designed for project demonstration, viva preparation, and quick reviewer walkthroughs. It summarizes the architecture, modules, sample credentials, business rules, and the exact value of the system.

Open Login Open Swagger Open Dashboard

Synopsis

Problem solved

Real estate teams often manage inventory, brokers, clients, and bookings across disconnected spreadsheets and messaging threads. This CRM centralizes property operations into one secure web application with role-based access, search, exports, analytics, and booking lifecycle tracking.

Technology

Stack used

  • Java 17 with Spring Boot and Maven
  • Spring Security with JWT authentication
  • Spring Data JPA with Hibernate ORM
  • MySQL for primary persistence
  • HTML, CSS, and JavaScript frontend served by Spring Boot
  • Swagger OpenAPI documentation and CSV/Excel exports

User Roles

Access model

  • Admin: full control over properties, clients, brokers, bookings, and exports
  • Broker: can work on inventory, clients, and booking operations through protected APIs
  • JWT tokens secure every protected backend request after login

Architecture

Layered backend design

Controller LayerExposes REST APIs for authentication, properties, clients, brokers, bookings, dashboard, and exports.
Service LayerContains business rules such as lead conversion, booking confirmation constraints, payment validation, inventory synchronization, and dashboard analytics aggregation.
Repository LayerUses JPA repositories and specifications for CRUD, filtering, pagination, and persistence.
Database LayerMySQL stores users, brokers, leads, properties, clients, and bookings with seeded Indian sample data.

Core Modules

Functional coverage

  • Dashboard analytics with lead funnel, booking conversion, collections, and city mix
  • Lead management with stage tracking, follow-up dates, source analysis, and one-click conversion to clients
  • Property management with filters by city, locality, type, price, status, and featured state
  • Client management with persisted preferred locality tracking
  • Broker management with linked login provisioning
  • Booking system with pending, confirmed, and cancelled states plus payment tracking
  • Export modules for leads, properties, clients, and bookings

Business Rules

Important logic enforced

  • Only one confirmed booking is allowed per property.
  • Confirmed bookings automatically mark the property as sold.
  • Removing or changing confirmed bookings re-syncs property status.
  • Payment status is derived and validated from booking amount and amount collected.
  • Leads can be converted into clients without re-entering buyer information.
  • Properties with active bookings cannot be deleted.
  • Brokers are backed by authenticated application users.

Viva Notes

Suggested explanation flow

  1. Start with the problem: real-estate data is usually scattered and manual.
  2. Explain the full-stack solution: Spring Boot backend, MySQL database, and integrated frontend.
  3. Show login and role-based access with JWT.
  4. Demonstrate lead capture, follow-up scheduling, and lead conversion into client records.
  5. Demonstrate inventory filtering, broker assignment, and booking creation with token/payment details.
  6. Highlight the automatic sold-status update after confirmed booking and payment validation rules.
  7. Show the dashboard analytics and export features as advanced functionality.

Possible Viva Questions

Prepared answers

Why layered architecture?It separates API handling, business logic, and persistence, which improves readability, testing, and maintainability.
Why JWT?JWT supports stateless authentication and fits REST APIs well because every request can be authenticated independently.
Why JPA/Hibernate?It reduces boilerplate SQL, maps Java entities to tables, and supports repository abstractions, filtering, and relationships.
How is data integrity maintained?Validation rules, foreign-key relationships, and service-layer business checks prevent invalid bookings and inconsistent property status.
Why add leads before clients?In real CRMs, an enquiry appears first as a lead. Converting leads to clients preserves sales pipeline visibility before a formal booking is created.
